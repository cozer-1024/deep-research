## 🚫 When NOT To Use
- 用户需要的是金融/投资专属分析（行情、板块、选股）→ 用 `investment-advisor` 或 `stock-picker-pipeline`
- 用户只需要简单的事实查询（查个定义、看个数字）→ 直接回答无需深度调研
- 用户只需要获取最新新闻/热点 → 用 TrendRadar 工具
- 调查内容仅限于获取单个 URL 的内容 → 用 `web_fetch` 即可

## 🔗 Related Skills
- **investment-advisor**: 金融领域专精的深度调研（事件驱动+产业链传导），deep-research 是通用框架不限领域
- **trendradar**: 热点监控，deep-research 做纵深调研而非横向监测
  零外部依赖，仅用 OpenClaw 内置工具（web_search / web_fetch / sessions_spawn）。
  
  SearXNG 集成：web_search 已自动路由到 SearXNG（当配置为搜索提供商时），
  实现多引擎聚合搜索（Google + Bing + DuckDuckGo 等）。
  注意：不要用 web_fetch 直接调 SearXNG API（被内网安全策略拦截），
  正确用法是直接使用 web_search 工具。

  触发条件（触发此 skill 的时机）：
  - 用户要求"深度研究"、"深度调研"、"deep research"、"deep dive"
  - 用户问需要多方查证才能回答的复杂事实性问题
  - 用户问"分析一下...的趋势/影响/前景"
  - 用户问"帮我调研一下..."、"研究一下..."
  - 用户说"写个行业分析/竞品分析/技术报告"
  
  不触发（用普通搜索即可）：
  - 简单事实查询（"今天天气"、"XX是什么"）
  - 用户明确说"简单搜一下"
  - 日常对话

# 深度调研技能（Deep Research）

## 框架：迭代深度研究框架（Iterative Deep Research Framework, IDRF）

```
┌─────────────────────────────────────────────────────────────────┐
│  Phase 0: 目标定义 & 范围切割                                    │
│  └─ 主题 → 子问题树 → 优先级 → 研究边界                          │
├─────────────────────────────────────────────────────────────────┤
│  Phase 1: 并行探索（Parallel Exploration）                       │
│  └─ 每个子问题 spawn 一个 sub-agent 独立搜索                      │
│     └─ 问题A ─→ [sub-agent A] ─→ 发现 + 来源                     │
│     └─ 问题B ─→ [sub-agent B] ─→ 发现 + 来源                     │
│     └─ 问题C ─→ [sub-agent C] ─→ 发现 + 来源                     │
├─────────────────────────────────────────────────────────────────┤
│  Phase 2: 交叉验证 & 深度挖掘（Cross-Reference & Deep Dive）      │
│  └─ 合并发现 → 冲突检测 → 追查矛盾点 → 补搜                      │
├─────────────────────────────────────────────────────────────────┤
│  Phase 3: 综合输出（Synthesis）                                  │
│  └─ 结构化报告 → 结论 + 论据 + 置信度 + 来源引用                  │
├─────────────────────────────────────────────────────────────────┤
│  Phase 4（可选）: 迭代深化（Iteration）                          │
│  └─ 识别知识盲区 → 新一轮研究 → 更新报告                         │
└─────────────────────────────────────────────────────────────────┘
```

## 核心原则

### 1. 先拆再搜
不要直接搜一个宽泛的问题。先拆成 3-7 个可搜索的子问题，并行处理。

### 2. 多源交叉
每条关键信息至少要有 2 个独立来源交叉验证。来源包括：搜索结果页、正文页、RSS、趋势数据。

### 3. 置信度标注
每条结论标记置信度：
- 🟢 高：多个可靠来源一致
- 🟡 中：来源有限但有据可查
- 🔴 低：单来源或间接推断，需进一步验证

### 4. 搜索通道优先策略（SearXNG First）

**SearXNG** 是聚合搜索引擎（同时搜 Google、Bing、DuckDuckGo 等多个引擎），比单个搜索引擎覆盖更广。

#### 检测流程

OpenClaw 的 `web_search` 工具如果配置了 SearXNG 作为搜索提供商，其返回结果的 `provider` 字段会显示 `searxng`。

```
检查 web_search 返回结果中的 provider 字段
├─ "provider": "searxng" → SearXNG 已就绪，正常使用 web_search 即可
└─ 其他 provider → 按默认搜索流程执行
```

> ⚠️ **不要** 尝试用 `web_fetch` 直接访问 SearXNG API（如 `http://localhost:8888/search?q=...`），
> Agent 工具安全策略会拦截对内网 IP 的请求，导致请求失败。正确的用法是直接走 `web_search`
> 工具，它会自动路由到已配置的 SearXNG 实例。

#### 工作原理

当 `web_search` 的 provider 为 `searxng` 时：
- 你发起 `web_search(query="关键词")` → OpenClaw 调用 SearXNG → SearXNG 聚合多个搜索引擎
- 返回的结果已包含多引擎去重后的内容，每条结果带标题、URL、摘要
- 等同于一次拿到 Google + Bing + DuckDuckGo 等多引擎的结果

#### 注意事项

- SearXNG 的某些后端搜索引擎可能不可用，导致部分关键词搜不到结果。遇到空结果时，**换关键词角度重试**。
- SearXNG 默认端口 `8888`，如果用户的 SearXNG 在其他端口，需在 OpenClaw 配置中修改 `search.provider` 相关设置。
- 不需要手动指定 categories/engines 参数，`web_search` 会使用默认配置。

#### 优势

- **多引擎聚合**：一次搜索拿到多个搜索引擎的结果
- **广度优先**：适合 Phase 1 的宽泛探索
- **去重**：SearXNG 自动合并相似结果
- **私密性**：不走商业搜索 API，无追踪

### 5. 分层搜索策略

```
第一层：web_search（自动走 SearXNG）→ 聚合多引擎，获取概览
第二层：web_fetch                  → 读正文，获取细节
第三层：TrendRadar                 → 看热度趋势、情感分析
第四层：sessions_spawn             → 并行深入探索子方向
```

**注意**：不存在 SearXNG 和 web_search 的二选一问题——web_search 已经自动通过 SearXNG 路由。

## 工作流程（Agent 执行指引）

### Phase 0：目标定义 & 范围切割

**目标**：把模糊的问题变成一组清晰的、可搜索的子问题。

**步骤**：
1. 理解用户的研究目标
2. 分析主题涉及的知识域
3. 拆解成 3-7 个子问题（每个问题对应一个独立搜索方向）
4. 确定优先级（必须回答 vs 锦上添花）
5. 明确研究边界（时间范围、地域范围、行业范围）

**输出**：研究计划（子问题列表 + 优先级 + 范围定义）

```
示例拆解：「分析 AI 编程助手的市场格局」→
  P0: 头部玩家有哪些？各自市场份额？
  P0: 各产品核心能力对比（功能、定价、模型）
  P1: 近 3 个月的重要动态/新产品
  P1: 开发者采用率和满意度
  P2: 技术趋势（Agentic coding、SWE-bench 等）
```

### Phase 1：并行探索

**目标**：用 sub-agent 并行搜索每个子问题，最大化信息获取效率。

**步骤**：
1. 对每个子问题执行第一层搜索（web_search，已自动走 SearXNG）
2. 对关键结果做第二层抓取（web_fetch）
3. 复杂子问题 spawn sub-agent 独立完成搜索-阅读链条

> **搜索技巧**：
> - 部分 SearXNG 后端引擎可能不可用，遇到空结果时**换关键词角度重试**
> - 同一个问题用中英文各搜一次，尤其中文主题搜索效果更好

**sub-agent 调用模板**（用于 Phase 1 的并行探索）：

```
task: >
  你是一个研究代理。请完成以下研究任务：
  
  研究问题：[子问题]
  背景上下文：[简要描述整体研究背景]
  
  请：
  1. 使用 web_search 搜索相关结果（至少搜 3 个不同角度的关键词）
     - 如果某个关键词搜到空结果，换一个关键词重试
     - 中英文关键词各尝试一下
  2. 使用 web_fetch 阅读至少 2 个关键来源的正文
  3. 整理发现，标记每条信息的来源 URL
  4. 对每条结论标注置信度（高/中/低）
  
  输出格式：
  ## 研究发现：[子问题]
  
  ### 关键发现
  - [发现1] 🟢/🟡/🔴 — 来源：[URL]
  - [发现2] ...
  
  ### 未解决的问题
  - 哪些问题还需要进一步探索
```

### Phase 2：交叉验证 & 深度挖掘

**目标**：合并所有发现，识别矛盾、填补空白。

**步骤**：
1. 合并 Phase 1 所有输出
2. 冲突检测：同一事实的不同说法标记为冲突
3. 空白识别：哪些子问题信息不足？
4. 深度追查：对冲突点和空白点追加搜索/抓取
5. 置信度调整

**注意**：矛盾不一定要强行统一，可以呈现多观点让用户判断。

### Phase 3：综合输出

**目标**：结构化研究报告，清晰呈现研究过程和结论。

**输出格式**：

```markdown
# 深度研究报告：[主题]

> 研究时间：YYYY-MM-DD | 研究方法：迭代深度研究框架

## 执行摘要

[300 字以内的核心结论]

## 研究框架

本次研究覆盖的子问题：
1. [子问题 A] — 优先级 P0
2. [子问题 B] — 优先级 P0
3. [子问题 C] — 优先级 P1

## 关键发现

### [子问题 A]

**结论**：[一句话结论] 🟢/🟡/🔴

**论据**：
- [论据 1]（来源：[URL]）
- [论据 2]（来源：[URL]）

**分析**：[你的分析和推理]

### [子问题 B]
...

## 交叉验证

- 高置信度结论：
- 存在争议的结论：
- 信息不足的结论：

## 研究局限

- 未覆盖的领域
- 数据时效性
- 潜在的偏见来源

## 参考资料

1. [标题](URL) — 发布日期
2. ...

## 后续建议

- 如需深挖哪些方向
- 建议关注什么时间点/事件
```

### Phase 4（可选）：迭代深化

**适用场景**：
- 研究发现了值得深挖的新方向
- 用户对初步报告不满意
- Phase 2 识别出大量信息空白

**步骤**：
1. 基于 Phase 3 输出，识别新的研究问题
2. 回到 Phase 1，对新问题进行并行探索
3. 更新报告，标注迭代版本

## Sub-Agent 故障处理

Sub-agent（sessions_spawn）可能偶发执行失败，原因通常包括：
- 模型推理超时
- 搜索工具返回异常
- 上下文过长

### 故障重试策略

```
sub-agent 返回失败 →
  1. 检查失败 session 的 history（sessions_history），判断是否部分结果可用
  2. 如果完全无结果：重新 spawn 一个 sub-agent，提示词加上更具体的搜索关键词
  3. 如果部分结果可用：手动补搜缺失的部分，不重新 spawn
  4. 连续失败 2 次：改在主会话中用 web_search 手动搜索，不在子代理中消耗 tokens
```

### 故障不影响整体
单个 sub-agent 失败不应阻塞整个研究。在主报告中标注该子问题状态为「⚠️ 需补搜」，其他子问题照常输出。

## 研究质量检查清单（Post-Research Checklist）

执行完研究后，自查：

- [ ] 每个子问题至少搜了 3+ 关键词角度
- [ ] 每条关键结论有 2+ 独立来源交叉验证
- [ ] 置信度标注完整（高/中/低）
- [ ] 所有来源有可访问的 URL
- [ ] 识别并标注了潜在的偏见/利益冲突来源
- [ ] 呈现了多观点，不强行求同
- [ ] 明确标注了信息盲区
- [ ] 报告结构完整（执行摘要 + 发现 + 验证 + 局限 + 参考）
- [ ] sub-agent 失败的不阻塞整体，标注「需补搜」

## 与 Deer-Flow 的能力对标

| 能力 | Deer-Flow | Deep Research Skill |
|------|-----------|-------------------|
| 问题拆解 | ✅ 自动 | ✅ SKILL.md 指导拆解 |
| 并行搜索 | ✅ sub-agent | ✅ sessions_spawn |
| 搜索引擎 | ✅ InfoQuest | ✅ SearXNG（多引擎聚合）+ web_search 双通道 |
| 网页抓取 | ✅ InfoQuest/搜索 | ✅ web_fetch |
| 代码执行 | ✅ 沙箱 | ⚠️ 依赖 exec，无沙箱 |
| 长时任务 | ✅ 小时级 | ⚠️ 需人工监督，受模型上下文限制 |
| 记忆持久化 | ✅ 数据库 | ⚠️ 文件持久化 |
| 可复用 | ✅ MCP/Skill | ✅ 本 SKILL.md |

**限制说明**：
- 无沙箱代码执行，数据分析需在 Agent 对话中完成
- 长时任务受模型上下文窗口限制，建议分段执行
- 报告通过文件写入或对话输出，而非 API
